# 发现的 Bug 详情

## BUG-01：世界观/人物关系画布撤销栈可能丢失编组操作
- **位置**：`WorldviewPanel.tsx` / `CharactersModule.tsx` 的 `useUndoRedo`
- **测试项**：4.15 撤销/重做
- **描述**：撤销栈基于快照（snapshot）机制，但编组操作（Ctrl+G 创建编组）修改的是 `worldviewGroups` / `charGroups` state，而非画布节点/边。如果编组操作没有被纳入快照，撤销时编组状态不会回滚。
- **严重程度**：低
- **影响**：编组操作后 Ctrl+Z 不会解散编组

## BUG-02：角色快照年龄解析为 NaN 时排序异常
- **位置**：`context-engine.ts:790`
- **测试项**：5.9 角色快照浏览
- **描述**：`parseInt(b.age) - parseInt(a.age)` — 如果 AI 生成的快照 age 字段为空字符串或非数字，`parseInt` 返回 `NaN`，排序结果不确定。虽然 `memory-updater.ts:304` 有 `parseInt(snap.age)` 做索引查找，同样有 NaN 风险。
- **严重程度**：低
- **影响**：快照排序可能乱序

## BUG-03：deleteChapter 读取闭包中的陈旧 chapters
- **位置**：`WritingModule.tsx:697-707`
- **测试项**：7.1.4 删除章节
- **描述**：`deleteChapter` 的 `useCallback` 依赖 `[pid, selectedChapterId, chapters]`。`chapters` 在依赖中意味着每次 chapters 变化都会重建回调，但 `chapters.find(c => c.id === chId)` 读取的是闭包快照。如果用户快速连续删除两个章节，第二次删除时闭包中的 `chapters` 可能还没更新。不过内部 `setChapters(prev => ...)` 使用了函数式更新，实际风险较低。
- **严重程度**：低
- **影响**：极端快速操作下可能删错章节标题提示

## BUG-04：AiWritingDialog 拖拽 handleMouseDown 依赖 pos 导致频繁重建
- **位置**：`AiWritingDialog.tsx:78-85`
- **测试项**：7.4.10 浮窗拖拽
- **描述**：`handleMouseDown` 的 `useCallback` 依赖 `[pos]`，每次拖拽移动都重建回调。虽然不影响功能，但拖拽时可能导致轻微卡顿。
- **严重程度**：低
- **影响**：拖拽体验轻微卡顿

## BUG-05：AI 写文安全超时清理不完整
- **位置**：`WritingModule.tsx:886-893, 999-1006`
- **测试项**：7.4.11 中途中止
- **描述**：安全超时 timer 在 `finally` 块中通过遍历 `timeoutIdsRef` 查找并清除。但如果 `finally` 执行时 timer 已经触发（5分钟超时），`clearTimeout` 对已触发的 timer 无效，不过由于 `aiWritingRef.current` 已在超时回调中重置，不会造成实际问题。但组件卸载时没有清理 `timeoutIdsRef` 中的 timer。
- **严重程度**：低
- **影响**：组件卸载后可能有孤立 timer 尝试 setState

## BUG-06：定稿流程无错误隔离，部分步骤失败可能导致后续步骤不执行
- **位置**：`WritingModule.tsx:1431-1514`
- **测试项**：7.5.1 完整定稿
- **描述**：定稿流程是一个 `onClick` 内的 async 函数，依次执行：保存 → 摘要 → 词条激活 → 角色预测 → 质量检查 → 备份 → 快照 → 新角色。虽然每个步骤有独立 try-catch，但 `saveContent()` 是同步调用，如果失败不会阻塞后续步骤。而 `updateMemory` 失败后，后续的 `activateNextChapterTerms` 仍会执行，但可能基于过期数据。更严重的是，这个 async 函数没有 `await` 保护，如果用户在定稿过程中切换章节，`selectedChapter` 可能变成 null。
- **严重程度**：中
- **影响**：快速切换章节时定稿可能写入错误章节

## BUG-07：故事圣经版本快照只存 metadata 不存实际内容
- **位置**：`StoryBibleModule.tsx` 版本记录功能
- **测试项**：8.10 版本记录
- **描述**：版本记录功能只保存时间戳和统计数据（词条数、角色数等），不保存风格指南/铁则的实际文本内容。用户无法回退到某个历史版本的实际内容。
- **严重程度**：低
- **影响**：版本记录功能实际用途有限

## BUG-08：素材库结构分析结果未持久化
- **位置**：`MaterialModule.tsx`
- **测试项**：10.10 分析结果持久化
- **描述**：AI 结构分析的结果存储在素材的 `structureAnalysis` 字段中，该字段在保存时会写入 localStorage。但需要确认保存逻辑是否在分析完成后自动触发。如果只存在 React state 中而没有调用 `saveItems`，刷新后会丢失。
- **严重程度**：中
- **影响**：用户可能丢失 AI 分析结果

## BUG-09：handleRegenerate 重新生成时删除了用户消息
- **位置**：`AiChatPanel.tsx:1112-1135`
- **测试项**：11.1.5 重新生成
- **描述**：`handleRegenerate` 删除最后一条 AI 回复**和**对应的用户消息（line 1124-1126），然后通过 `setInput` + `rAF` 点击发送按钮重新发送。问题：1) 用户消息被删除又重新添加，会导致消息顺序变化；2) 如果中间有 system 消息，`userMsg` 可能为 null，此时只删除 AI 回复不重新发送。
- **严重程度**：中
- **影响**：重新生成时可能丢失用户原始输入

## BUG-10：handleConfirmEdit 使用 setInput + rAF 可能读到旧 input 值
- **位置**：`AiChatPanel.tsx:1061-1077`
- **测试项**：11.1.8 编辑消息
- **描述**：`setInput(text)` 是异步 state 更新，`requestAnimationFrame` 中点击发送按钮触发的 `send` 函数读取 `input` state 时，可能还是旧值。虽然 `send` 函数在 `requestAnimationFrame` 回调执行时 React 可能已经 flush 了 state 更新（因为 rAF 在下一帧），但这不是保证行为。
- **严重程度**：中
- **影响**：编辑消息重新发送时可能发送旧内容

## BUG-11：语音输入未配置时的提示不够明确
- **位置**：`mock-backend.ts:601-603`
- **测试项**：11.3.3 未配置时录音
- **描述**：未启用语音时，`stt_transcribe` 返回的文本是 `"（语音识别未启用。请在设置-语音配置中开启并填入 API Key。）"`，这个文本会被直接填入输入框，用户可能误以为是识别结果。
- **严重程度**：低
- **影响**：UX 不够友好

## BUG-12：词条更新（X改为Y）时删除旧词条并创建新词条，丢失 ID 连续性
- **位置**：`AiChatPanel.tsx:1254-1256`
- **测试项**：11.4.7 更新词条
- **描述**：`api.deleteWorldTerm(target.id)` 后 `api.saveWorldTerm({ ...target, id: uuid(), title: newName })` — 创建了新 ID 的词条。这会导致：1) 关系边（edges）中引用旧 ID 的连线断裂；2) 编组中引用旧 ID 的归属丢失；3) 世界观画布上旧节点消失，新节点出现在默认位置。
- **严重程度**：高
- **影响**：词条改名后关系连线和编组丢失

## BUG-13：记忆压缩切片逻辑可能压缩不正确的消息范围
- **位置**：`memory-engine.ts:119-126`
- **测试项**：11.6.2 短期记忆压缩
- **描述**：`const processable = unprocessed.filter(m => m.role !== "system")` 过滤了 system 消息，但 `const rounds = Math.floor(processable.filter(m => m.role === "user").length)` 只计算 user 消息数量。然后 `const messagesToCompress = processable.slice(0, rounds)` 用 rounds 作为索引切片 — 但 rounds 是 user 消息数量，不是消息总数。如果 20 轮对话中有 20 条 user + 20 条 assistant = 40 条消息，`rounds=20`，但 `processable.slice(0, 20)` 只取前 20 条（大约前 10 轮），而不是全部 20 轮。
- **严重程度**：中
- **影响**：记忆压缩只压缩了一半的消息

## BUG-14：项目阶段（stage）永远不会自动更新
- **位置**：`app-store.ts` / 全局
- **测试项**：12.7 项目阶段显示
- **描述**：`Project.stage` 在创建时设为 `"ideation"`，但整个代码库中没有任何地方调用 `stage` 的更新逻辑。无论是定稿、锁定框架还是完成全书，`stage` 都不会变化。Header 始终显示"构思中"。
- **严重程度**：中
- **影响**：项目阶段指示器形同虚设

## BUG-15：导入数据后 novel-workbench-mock 键的恢复可能与 importProject 冲突
- **位置**：`SettingsModal.tsx:956-963` (doImportWithMode)
- **测试项**：2.4.5 导入后数据完整性
- **描述**：`doImportWithMode` 先调用 `api.importProject` 恢复项目数据（写入 mock store），然后遍历 `data.keys` 写入 localStorage。如果 keys 中包含 `novel-workbench-mock`（旧版导出格式），会用旧的完整 mock 覆盖刚刚 importProject 更新后的 mock store，导致导入的项目数据丢失。代码在 line 1025 有 `key !== "novel-workbench-mock"` 的过滤，但只在 `doImportWithMode` 中，不在 `handleImport` 的无项目数据分支中。
- **严重程度**：中
- **影响**：旧版格式导入可能丢失数据

## BUG-16：定稿后摘要写入 logStore 但上下文引擎从 api 读取，存在时序差异
- **位置**：`memory-updater.ts:540-548` vs `context-engine.ts:420-427`
- **测试项**：15.9 章节摘要→P1
- **描述**：`saveSummary` 直接写入 `novel-workbench-log-{pid}` 的 localStorage。而 `loadRecentSummaries` 调用 `api.getChapterSummaries`，在浏览器模式下 mock 后端也是读 localStorage，应该一致。但在 EXE 模式下，`api.getChapterSummaries` 走 Rust 后端读 SQLite，而 `saveSummary` 写的是 localStorage + 异步 SQLite。如果异步写入还没完成就读取，可能读不到最新摘要。
- **严重程度**：低
- **影响**：EXE 模式下定稿后立即切换章节，上下文可能不包含最新摘要

## BUG-17：角色状态更新同 BUG-16 的时序问题
- **位置**：同上
- **测试项**：15.10 角色状态→P3
- **描述**：同 BUG-16，角色状态写入 logStore 后，P3 上下文读取可能存在时序差异。
- **严重程度**：低

## BUG-18：EXE 模式下 localStorage 满时 setSync 静默失败，SQLite 异步写入可能丢失
- **位置**：`storage.ts:117-124`
- **测试项**：16.4 localStorage 满
- **描述**：`setSync` 中 `localStorage.setItem` 失败时 catch 静默跳过，然后异步调用 `api.setSetting` 写入 SQLite。但 `api.setSetting` 是 fire-and-forget（`.catch(() => {})`），如果 SQLite 写入也失败（如磁盘满），数据会丢失且无任何提示。
- **严重程度**：低
- **影响**：极端情况下数据静默丢失

## BUG-19：上下文面板 loadContextPanelData 竞态条件
- **位置**：`WritingModule.tsx:507-590`
- **测试项**：16.6 并发 AI 请求
- **描述**：快速切换章节时，多个 `loadContextPanelData` 并发执行。虽然有 `loadGen` 计数器在部分位置做了过期检查（line 564, 570, 575），但 `Promise.all` 返回后的 `setCtxSummaries`、`setCtxBeatCards` 等调用没有检查 `loadGen`，可能导致旧章节数据覆盖新章节数据。
- **严重程度**：高
- **影响**：快速切换章节时上下文面板显示错误数据
